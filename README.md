# 헬스장 챗봇 SaaS — 사장님 웹 포털 (MVP)

사장님이 로그인해서 **직접 회원을 업로드·관리**하는 멀티테넌트 웹 포털.
설계 기획서(`헬스장챗봇_사장님웹포털_설계기획서.md`)의 Phase 1을 실제 동작하는 앱으로 구현한 버전.

## 실행 방법

```bash
cd gym-portal
npm install
npm start
```
브라우저에서 **http://localhost:4000** 접속 →
- 데모 로그인: **demo@demo.com / demo1234**
- 또는 "지점 등록"으로 새 계정(=새 지점) 생성

> 포트 변경: `PORT=5000 npm start`

## 들어있는 기능 (MVP)

| 메뉴 | 내용 |
|---|---|
| **로그인/지점등록** | 이메일+비밀번호(bcrypt), 세션. 계정=지점 |
| **대시보드** | 오늘 출석·신규·매출·만료임박·휴면·PT소진임박 + 관리필요 회원 |
| **회원 관리** | 목록·검색, **CSV 업로드(전화번호 기준 추가/갱신)**, 개별 편집/삭제, 샘플 CSV |
| **PT 회원** | 이용권 잔여·담당강사·유효기간, 소진임박 표시, 세션기록(완료 시 잔여 자동 차감) |
| **리포트** | 주간/월간 요약(신규·매출·방문·PT·상담/요청·관리필요) |
| **접수함** | 챗봇으로 들어온 상담(리드)·각종 요청 상태 처리 |
| **매장 설정** | 매장명·가격·강사·공지·이벤트 편집 (챗봇 응답에 반영될 값) |
| **발송 관리** | 자동발송 항목·발송 스위치(dry-run↔실발송)·발송 이력 |

## 핵심 설계

- **멀티테넌트**: 모든 데이터가 `gym_id`로 분리됨. 로그인한 지점 데이터만 접근(테넌트 격리 검증 완료).
- **영속화**: `data/db.json` 파일에 저장 → **서버를 재시작해도 데이터가 유지됨**(재시작 시 초기화되던 문제 해결).
  - 데이터 초기화: `data/` 폴더 삭제 후 재실행하면 데모 데이터로 다시 생성.
- **CSV 규격**(헤더): `전화번호,이름,회원권종류,만료일,가입일,PT총회,PT잔여,PT담당강사,락커여부,락커만료일,메모`
  - 포털의 "샘플 CSV 내려받기"로 양식 확인 가능. 전화번호 기준 upsert(기존이면 갱신).

## ✅ 챗봇 연동 (완료)

이 포털은 **카카오 챗봇 스킬 엔드포인트(`/skill/*`)를 같은 서버·같은 DB로 제공**합니다.
즉 사장님이 포털에서 올리거나 수정한 데이터로 챗봇이 그대로 응답합니다. (멀티테넌트)

- `POST /skill/welcome` · `membership` · `pt` · `faq` · `trainer` · `notice` · `event` · `lead` · `request`
- **지점 식별**: 카카오 요청의 `bot.id` → `bots` 테이블에서 `gym_id` 매핑 (없으면 첫 지점으로 폴백)
- **읽기**: 회원권/PT 현황은 그 지점 실 회원, 가격·강사·공지·이벤트는 매장 설정에서
- **쓰기(양방향)**: 챗봇 상담신청·각종요청 → 포털 **접수함**에 자동 기록
- 검증 완료: 실 회원 조회, PT 소진임박 감지, 상담신청 write-back, **같은 전화번호라도 봇이 다르면 다른 지점 회원 반환(테넌트 격리)**

### 실서비스 전환(go-live)
1. 이 포털 앱을 **상시 가동 호스팅에 배포**(데이터는 영구 디스크 또는 PostgreSQL).
2. 각 지점 오픈빌더 봇의 스킬 URL을 **이 서버의 `/skill/*`** 로 지정.
3. 포털 `bots`에 `kakao_bot_id ↔ gym_id` 등록.
4. 카카오 채널 연결 + 알림톡 템플릿 승인 → 발송 스위치 ON.

> 현재 데모 봇(`6a2ebca4e4f43f5dd57865cd`)은 데모 지점에 매핑되어 있습니다.

## 다음 단계 (기획서 참고)

1. **운영 전환** — 데이터 계층(`db.js`)을 PostgreSQL로 교체, 상시 가동 호스팅 배포.
2. **실 발송** — 카카오 채널 연결 + 알림톡 템플릿 승인 후 발송 스위치 ON, 리포트 자동 발송 cron 연결.
3. 개인정보·마케팅 수신 동의, 결제(PG), 지점 온보딩 자동화(블록/스킬 템플릿 복제).

## 파일 구조

```
gym-portal/
├─ server.js   # Express 라우트·인증·CSV 업로드
├─ db.js       # 데이터 계층(JSON 파일 · 운영 시 Postgres로 교체)
├─ views.js    # HTML 렌더(사이드바·대시보드·표·폼)
├─ data/db.json# 데이터(자동 생성, git 제외)
└─ package.json
```

> 참고: 이 MVP는 네이티브 의존성 없이 어디서든 바로 구동되도록 JSON 파일 저장을 씁니다.
> 운영 배포 시 `db.js`만 PostgreSQL 구현으로 교체하면 나머지 코드는 그대로 재사용됩니다.
